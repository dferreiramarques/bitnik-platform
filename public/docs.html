Bitnik Framework

A visual node editor for designing board games — from physical rulebook to playable digital prototype.

Bitnik is an open-source engine and visual design tool that lets game designers describe board game logic as a graph of connected nodes — without writing code. The node graph is exported as a .bge file that the runtime interprets directly, or that a code generator converts into a working boardgame.io project.

What Bitnik is

A domain-specific visual programming language for board games. You describe what happens — not how to implement it.

What Bitnik is not

A general game engine. It is purpose-built for turn-based, card, tile, and dice games — the patterns that appear in the BoardGameGeek top 1000.

Design philosophy

The taxonomy of 43 nodes was derived empirically by porting four real games — Bulbous (trick-taking), Capivaras (secret betting), Praia das Percebes (tile placement), and Nine Oils (dice combos) — and recording every pattern that required a new abstraction. The result covers ~95% of published board game mechanics.

Core concepts

Nodes

A node is an atomic game mechanic. Each node has a type (e.g. TURN_BLOCK), a label, zero or more input ports, and zero or more output ports. Nodes are grouped into 7 families:

FLOW LOGIC ACTIONS DATA EVENTS TRIGGERS ASSETS

Edges

An edge connects an output port of one node to an input port of another. Edges represent sequential flow ("after this, do that") or data passing. The graph of nodes and edges is the complete description of the game.

Triggers vs edges

Edges model sequential flow. Triggers (the pink family) model reactive behaviour — "whenever X happens, fire Y" — regardless of where the game currently is in the flow. A ON_DECK_EMPTY trigger fires even if the game is in the middle of a player action.

Swimlanes

Swimlanes are horizontal regions that organise nodes into phases: Setup (20%), Game Loop (60%), End Game (20%). They are cosmetic — they do not affect execution — but they enforce readability and map directly to the scenes concept in the runtime.

The Turn Block

The Turn Block is a macro-node that encapsulates the entire structure of a player turn. Instead of wiring dozens of individual action nodes, you define the constraint (choose 1, up to N, all mandatory, free), the available actions, and the possible outcomes. Double-clicking it opens the detail panel.

The .bge file

The exported JSON file contains the full graph: swimlanes, nodes (with type, position, and config), and edges. It is the portable, version-controllable description of your game. The runtime reads it directly; the code generator converts it to TypeScript.

Tutorial — building a game from scratch

This tutorial walks through building a simple 2–4 player resource collection game. You will model the full game loop in ~15 minutes.

Step 1 — Start with the Setup swimlane

1

Drop a Game Start node

Drag Game Start (FLOW, orange) from the palette into the Setup swimlane. In the Inspector, set Max jogadores to 4.

2

Add initial distribution

Drop a Draw Card node next to it. Set Amount to 5 (starting hand). Connect Game Start next → Draw Card in.

3

Add a Shared Board

Drop a Shared Board (DATA, purple) in Setup. Set Width and Height to 7 and Adjacency to hex. This is the board all players share.

Step 2 — Model the Game Loop

4

Drop a Turn Block

Drag Turn Block (FLOW) into the Game Loop swimlane. Connect Draw Card next → Turn Block start.

5

Configure the Turn Block

Double-click the Turn Block. Set Constraint to Até N, Max to 2. Add actions: Collect Resource (opc.), Trade (opc.), Place Token (opc.). Outcomes: end_turn, endgame.

6

Add victory check

Drop Check Victory (LOGIC). Connect Turn Block end_turn → Check Victory in. Connect Check Victory continue → Turn Block start (the loop). Connect win → the End Game swimlane.

Step 3 — End Game

7

Add scoring

Drop Spatial Score (EVENTS) in the End Game lane. Set Formula to resources_collected. Drop Game End (FLOW) and connect Spatial Score next → Game End prev.

8

Add endgame trigger

Drop On Deck Empty (TRIGGERS). Connect its fire port to Turn Block endgame. This fires whenever the deck runs out — wherever the game is in the flow.

Step 4 — Export

9

Validate and export

Click Validar to check for missing Game Start or Game End nodes. Click Exportar .bge — you get a meu_jogo.bge.json file ready for the runtime or code generator.

Tip: Name all your nodes clearly — the label becomes the function name in generated code. "Collect wood from adjacent hex" is better than "Collect Resource".

NodeDescription

Draw CardMoves N cards from a deck to a player's hand.
Swap ★Atomically discards N cards and draws N cards. Used for hand refresh mechanics.
Tile Place ★Places a tile on the Shared Board, with adjacency validation.
Move PieceMoves a piece from one Zone to another.

Steal ★Transfers a piece or resource from an opponent's zone to the active player.
Card Modifier ★A card played before a roll that modifies the outcome (e.g. Temptress in Nine Oils).

DATA — Game state containers

NodeDescription
DeckAn ordered collection of cards. Supports shuffle, draw, discard pile recycling.
HandA player's private card collection. Hidden from other players.
CounterA numeric value — score, resources, round number.
Player StateA player's full attribute set — name, colour, score, resources, pieces.
ZoneA named area on the board belonging to one player (e.g. stall, tableau).
Shared Board ★A grid (hex or square) shared by all players. Tracks cell contents and adjacency.
Structure ★An N-slot collectible objective. Fill all slots to complete it (e.g. Baelfungious).
Locked Slot ★A slot with three states: locked → open → filled. Unlocked by specific combos.
Role ★A rotating token (e.g. Governor, First Player) that passes between players each round.

EVENTS — Scoring and notifications

NodeDescription
Notify AllSends a public message to all players (e.g. "Player X played a card").
Notify PlayerSends a private message to one player only.
Score PointsAdds or subtracts points from a player immediately when triggered.
Spatial Score ★Computes end-game points based on board geometry (e.g. bathers in a lifeguard's line).
Majority ★Awards bonus points to the player with the most tokens in a completed structure.
Collection Bonus ★Awards bonus points for collecting N distinct types of an item.
First Claim ★Awards an objective to the first player who satisfies its condition.
Progressive Reveal ★Maintains a visible window of N objectives from a larger pool; reveals new ones as they are claimed.
Secret Bet ★All players secretly commit a bet simultaneously; bets are revealed together.
Competing Claim ★A unique bet wins; if two or more players bet the same, nobody wins.
Reactive Defense ★An attack action opens a response window for the opponent within the same turn.

TRIGGERS — Reactive listeners

Triggers are always-on listeners. Unlike edges (which model sequential flow), triggers fire whenever their condition is met, regardless of the current game phase.

NodeFires when…
On Turn StartAny player's turn begins.
On Turn EndAny player's turn ends.
On Round EndAll players have completed their turn in a round.
On Tile PlacedAny tile is placed on the Shared Board. Config: filter by tile type.
On Card PlayedAny card is played from hand. Config: filter by card type.
On Score ChangeAny player's score changes. Config: optional minimum threshold.
On Piece PlacedAny piece is placed on the board.
On Resource GainA player gains a resource. Config: filter by resource type.
On Deck EmptyA deck runs out of cards (after discard recycling).
On Condition ★A custom condition becomes true. Config: expression + repeat policy (always / once / per-turn).
On Player ActionA player performs any action. Config: filter by action type.
After Effect ★Fires after a specified source node completes. Config: optional delay in ms.

ASSETS — Visual and audio presence

NodeDescription
Board LayoutConfigures the visual board — hex, square, or custom. Links to the Shared Board data node.
Piece VisualAssigns a sprite or colour to a piece type, per player colour.
Card TemplateDefines the front/back layout of a card with dynamic fields.
AnimationPlays a named animation (move, flip, glow) when triggered by an event.
Sound FXPlays a sound when triggered.
Timer ★Runs a countdown. On expiry, fires the done port (used for Tie Break, speed rounds).
Simultaneous ★Holds the flow until all players (or a quorum) have completed an action in parallel.

Turn Block — in depth

The Turn Block is the most important node in the framework. It replaces the need to wire a Turn Start → action nodes → Turn End sequence for every game.

Constraint modes

ModeBehaviour
Escolhe 1Player picks exactly one action from the list.
Até NPlayer may perform up to N actions in any order (default: 2).
Todas obr.Player must perform all listed actions in order.
LivrePlayer may perform any number of any action in any order.

Actions

Each action in the list is an action node type from the ACTIONS family. Mark it as obrig. (mandatory) or opc. (optional). The Turn Block enforces this at runtime — a player cannot end their turn without completing mandatory actions.

Outcomes

Outcomes become output ports on the node. Connect them to the next node in the flow. Common outcomes:

  • end_turn → loops back to the next player's turn
  • endgame → jumps to the End Game swimlane
  • tie → routes to a tiebreaker node

Tip: You can add as many outcomes as your game needs. Each outcome generates a distinct output port and a distinct transition in the generated state machine.

Swimlanes

Swimlanes divide the canvas into three horizontal phases. The default proportions are 20 / 60 / 20 — Setup, Game Loop, End Game.

Operations

ActionHow
CollapseClick the ▼ button in the swimlane header. Nodes inside become hidden. The lane collapses to a slim bar.
ExpandClick ▶ on a collapsed lane.
ResizeDrag the thin handle at the bottom edge of each lane. The percentage indicator updates live.
RenameClick the name in the header and type.

Note: Swimlanes are cosmetic — they do not affect execution order. The graph edges and triggers define the actual runtime flow.

.bge file format

The exported file is a plain JSON object. It is the single source of truth for your game design.

{
  "_format": "bge",
  "_version": "0.4",
  "game": { "name": "meu_jogo" },

  "swimlanes": [
    { "id": "sl1", "name": "Setup",     "color": "#f5a623", "y": 0,    "h": 360 },
    { "id": "sl2", "name": "Game Loop", "color": "#2ec4a4", "y": 360,  "h": 1080 },
    { "id": "sl3", "name": "End Game",  "color": "#f87171", "y": 1440, "h": 360 }
  ],

  "nodes": [
    {
      "id": "n1",
      "node": "GAME_START",
      "label": "Game Start",
      "x": 60, "y": 55,
      "config": { "maxPlayers": 4 }
    },
    {
      "id": "n2",
      "node": "TURN_BLOCK",
      "label": "Turno do Jogador",
      "x": 280, "y": 415,
      "config": {
        "tb": {
          "constraint": "up_to_n",
          "max": 2,
          "actions": [
            { "id": "a1", "type": "Collect Resource", "mand": false },
            { "id": "a2", "type": "Place Token",       "mand": false }
          ],
          "outcomes": [
            { "id": "o1", "label": "end_turn", "color": "#4ade80" },
            { "id": "o2", "label": "endgame",  "color": "#f87171" }
          ]
        }
      }
    }
  ],

  "edges": [
    { "id": "e1", "fromNode": "n1", "fromPort": "next",
                  "toNode":   "n2", "toPort":   "start" }
  ],

  "engine_nodes_used": ["GAME_START", "TURN_BLOCK", "GAME_END"]
}

Code generation

The .bge file can be fed into the Bitnik code generator, which produces a ready-to-run boardgame.io TypeScript project.

Output files

FileContents
src/types/index.tsTypeScript interfaces for all game entities derived from the DATA nodes.
src/game/game.tsThe boardgame.io Game definition — state machine, moves, turn config, endIf.
src/game/game.config.tsAll configurable values from node inspector fields (max players, hand sizes, etc.).
src/utils/Helper utilities — hex grid math, deck management, scoring functions.

Status: The code generator is in development. The .bge format is stable — files created today will be importable by the generator when released.

Manual porting workflow

Until the code generator ships, use the .bge file as a design specification. The Bulbous port (included in the repository) shows the mapping between graph nodes and boardgame.io code:

GAME_START     → setup() function
TURN_BLOCK     → turn.moves + turn.onBegin/onEnd
CHECK_VICTORY  → endIf()
TRIGGERS       → onBegin/onEnd hooks + event listeners
SIMULTANEOUS   → activePlayers configuration
TIMER          → setTimeout + tieBreakTimeout
SECRET_BET     → hidden state + playerView filter
